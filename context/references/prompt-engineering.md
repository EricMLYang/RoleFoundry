# Skill 設計的 Prompt 工程最佳實踐

> 如何撰寫讓 Agent 產出穩定、高品質結果的 SKILL.md。

---

## 核心原則

### 1. 範例比指令更重要

Agent 透過 few-shot 學習從範例推斷期望的輸出格式和品質。
一個有 2 個範例的 SKILL.md 產出品質，通常優於一個有詳細指令但無範例的 SKILL.md。

**行動**：在 examples/ 中至少提供 1 組輸入/輸出對，越真實越好。

### 2. 步驟越清楚，幻覺越少

模糊的指令（「分析需求」）比具體的步驟（「列出 3 個最常見的使用場景，每個附觸發條件」）更容易產生幻覺。

**行動**：每個流程步驟都說清楚「做什麼 + 產出什麼格式 + 達成什麼條件繼續」。

### 3. 模式標記讓 Agent 知道何時等待

`[interactive]` 告訴 Agent 要停下來等使用者輸入；`[autonomous]` 告訴 Agent 可以自主決定。
沒有標記 → Agent 可能在不必要的地方停下來，或在應該確認的地方衝過去。

---

## 常見失敗模式

| 失敗模式 | 症狀 | 修復方式 |
|---------|------|---------|
| 觸發條件太模糊 | Agent 不知道何時用這個 Skill | 改成「使用者說 X 或 Y 的句子時」|
| 流程步驟無標記 | Agent 在錯誤的地方等待或繼續 | 每個步驟加 [interactive/autonomous/hybrid] |
| 品質標準主觀 | 無法客觀判斷是否通過 | 改成可計數或可 yes/no 判斷的條件 |
| 缺少 Limitations | Agent 過度承諾 | 誠實說明 Skill 做不到什麼 |
| examples/ 是空的 | 產出格式每次不一致 | 在真實任務後補充 1-2 個範例對 |

## Skill 長度建議

| 部分 | 建議長度 |
|------|---------|
| 用途（Purpose） | 3-5 行 |
| 何時使用 | 2-4 個觸發句 |
| 流程步驟 | 3-7 個步驟，每步驟 3-8 行 |
| 品質標準 | 3-6 條 checkbox |
| 已知限制 | 2-4 條 |
| **總計** | **50-150 行（超過考慮拆分）** |

---

## 觸發條件的寫法

**好的觸發條件**（使用者真實說的話）：
- 「幫我建立一個新的角色 Repo」
- 「我想鍛造一個架構師的 Expert Repo」

**不好的觸發條件**（系統性描述，使用者不會這樣說）：
- 「當 role-bootstrap 的前置條件滿足時」
- 「需要進行角色身份設計工作流程時」

**原則**：把自己代入使用者，他真實說出的話才是好的觸發條件。
