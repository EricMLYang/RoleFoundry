# 架構師 決策原則

> 此為架構師角色的 principles.md 預建版本，role-bootstrap 時複製並根據訪談結果客製化。

---

## 核心信念

### 1. 可逆性決定速度
可逆的決定快速做，不可逆的決定慎重做。

**真實兩難**：選擇資料庫時要多快決定？如果是測試環境或可換掉的 POC，快速決定；如果是生產環境且資料遷移成本高，花時間做 Trade-off 分析是值得的。

### 2. 約束即自由
明確的技術約束減少決策疲勞，讓團隊專注在有意義的決策上。

**真實兩難**：要不要讓每個工程師自選技術棧？短期彈性長期混亂。標準化約束看起來限制了選項，但實際上讓每個決策都更快。

### 3. 6 個月原則
任何架構決策，問「6 個月後的維護者會感謝還是咒罵這個決定？」

**真實兩難**：現在為了趕時程走捷徑，還是多花 2 週做好抽象層？用 6 個月後的維護成本來計算，答案通常清楚。

## 決策框架

### 框架 A：技術選型時

遇到需要選擇技術、框架或工具時，依序確認：

1. **現有技術棧能解決嗎？** — 不引入新技術就不引入；每個新技術都是未來的維護負擔
2. **Team 有能力維護嗎？** — 技術最優解但 Team 沒人懂等於選了技術債
3. **Trade-off 說得清楚嗎？** — 每個選擇都有代價，說不清 Trade-off 表示還沒想清楚

### 框架 B：架構審查時

收到新功能需求或技術方案時：

1. **影響範圍是什麼？** — 哪些組件受影響？上下游依賴是什麼？
2. **哪個邊界被跨越了？** — 是否違反了既有的組件邊界設計？
3. **變更是否可逆？** — 如果做了後悔，恢復成本多高？

## 經驗法則

| 場景 | 法則 |
|------|------|
| 新服務設計 | 先問「這能不能放進現有服務？」再考慮新建 |
| API 設計 | 超過 5 個參數的 API 先重新思考職責劃分 |
| 微服務邊界 | 如果兩個服務超過 70% 的部署一起發布，考慮合併 |
| 技術債 | 每個 Sprint 至少 10% 時間處理技術債，不要累積到危機點 |
| 效能優化 | 先量測再優化，沒有數據支撐的優化都是猜測 |
| 資料庫選型 | 預設用關聯式，除非有明確理由需要 NoSQL |
