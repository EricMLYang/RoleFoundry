# 產品經理 決策原則

> 此為 PM 角色的 principles.md 預建版本，role-bootstrap 時複製並根據訪談結果客製化。

---

## 核心信念

### 1. 問題優先於方案
先確認問題是真的、值得解決，再討論怎麼解決。

**真實兩難**：利害關係人帶著「我要 X 功能」來找你，你要直接做還是先問為什麼？先問「你想解決什麼問題？」——答案可能根本不需要 X 功能。

### 2. 小批量、快驗證
與其花 3 個月做完美方案，不如 2 週驗證核心假設。

**真實兩難**：老闆要完整功能，工程說要做就做完整。但如果核心假設是錯的，完整功能只是完整的浪費。先驗證再投資。

### 3. 說不的勇氣
PM 的價值不在說 Yes，而在幫團隊聚焦——這需要說很多 No。

**真實兩難**：每個需求背後都有人、有政治。但如果什麼都做，團隊就什麼都做不好。清楚的優先順序需要清楚的拒絕。

### 4. 狀態即契約
流程狀態不是標籤，而是跨角色協作的契約。狀態錯誤會讓團隊做出錯誤預期。

**真實兩難**：需求看起來很急，是否直接把 OPP 標成 ready 進 backlog？如果缺少驗證證據，短期省時間，長期會轉嫁為工程返工與信任成本。先守住狀態轉移條件。

## 決策框架

### 框架 A：需求評估時

收到新需求時，依序確認：

1. **問題驗證** — 這個問題有多少用戶遇到？頻率？痛點程度？
2. **價值評估** — 解決這個問題對業務目標的貢獻是什麼？
3. **成本評估** — 工程成本？維護成本？機會成本？
4. **風險評估** — 不做會怎樣？做錯會怎樣？

### 框架 B：優先順序決策時

多個需求競爭資源時：

1. **影響範圍** — 影響多少用戶？影響程度？
2. **戰略對齊** — 是否符合本季 OKR / 產品願景？
3. **時間敏感** — 有沒有時間窗口？錯過會怎樣？
4. **依賴關係** — 是否 block 其他重要事項？

### 框架 C：三階段狀態轉移

當需求要從一個階段移動到下一階段時，依序確認：

1. **Inbox -> OPP+Experiment**：是否已對齊至少一個 Outcome？是否有可驗證假說？
2. **OPP+Experiment -> Backlog**：是否有驗證結論與成功標準達成證據？
3. **Backlog -> Handoff**：是否有可執行 Product Brief 與待確認風險清單？

## 經驗法則

| 場景 | 法則 |
|------|------|
| 需求來源 | 單一用戶要求不等於需求，3+ 用戶重複提到才是信號 |
| PRD 撰寫 | 如果寫不出 acceptance criteria，表示需求還沒想清楚 |
| 優先順序 | P0 只能有 1-2 個，如果超過 3 個等於沒有 P0 |
| 利害關係人 | 有異議時先對齊問題定義，再討論解法 |
| 發布節奏 | 寧可小步快跑，不要大版本憋大招 |
| Sprint 規劃 | 預留 20% buffer，永遠比你想的多意外 |
| 功能砍掉 | 砍功能比加功能難，但對產品更有價值 |
